# 优化指导

本文提供一些可能的优化方向，用于作为优化参考。注意，一切优化以具体case，实际的camodel仿真为主。

## VF概念

VF 是昇腾A5代际由regbase带来的新的vector计算编程范式，其核心思想为将需要计算的一段数据(一次可计算256B的数据，即64个FP32数据或128个FP16数据。)由UB (unified buffer 248kB) 提取到UB与vector核间的寄存器内，经过一系列vector计算后，再搬回到UB中，减少了从UB到vector核内的反复搬运开销。在代码实现上由 `__VEC_SCOPE__`对VF结构体进行包裹，一般包含了`vlds` (将数据从UB搬运到寄存器中)， `vadds, vadd, vexp ...`等计算指令，`vsts`(将数据从寄存器搬运到UB中)

## VF高性能计算

A5 代际的高性能计算核心主要在于如下几点：
- 减少反复UB数据搬运，可在寄存器中进行批量vector计算；
- 使能双发射，A5代际采用了双EXU (executio unitary) 单元，除了少数规约类指令只能在EXU0单元计算，其余指令均可在EXU0与EXU1单元指令计算，尽量使能双发射是VF高性能计算的核心；
- 使能hardware loop计算。为使能hardware loop计算，loop内的迭代变量要求为uint16_t，loop bound为uint_16t，loop迭代采用规范化形式，即迭代变量从0开始，每次迭代需要加1。Hardware loop最多支持4层嵌套循环，不允许有跳转指令，如if/else。

## VF优化方向

### VF拆分与合并

尽量将VF进行融合，即多个`__VEC_SCOPE__`结构体合并为一个`__VEC_SCOPE__`，此操作可被称之为VF浅融合，浅融合的收益点在于节省了VF头尾开销，阻碍点在于增大了UB压力。因此需要在UB空间允许的情况下尽量使能VF融合。

### loop unroll

ccec 编译器支持loop unroll功能，可通过在for循环前加入`#pragma unroll(N)`来使能N次unroll展开，编译器unroll选项仅支持最内层循环的unroll展开，展开形式为将循环体内指令ABC变为AABBCC的形式。Unroll展开的收益点在于将没有依赖关系的两次循环指令发在一起，尽可能使能双发射，可能的负收益点在于增大寄存器压力，以及当有长短指令混合的情况时，可能会增加某段区间内长指令密度，降低该区间内的IPC，可能的解决方法为在展开后进行指令重排序，在长指令密集区域插入无依赖的短指令，降低长指令密度。同时，通过`#pragma unroll(N)`进行unroll展开可能会遇到编译器不支持的情况，这时可通过手动unroll来避免此情况。

### loop 切分

#### loop横向切分

对于拥有长依赖链的loop结构体，过长的依赖链会导致性能瓶颈，原因主要是consumer指令需要等待producer指令计算后过forwarding时间才能开始计算，极大降低了IPC，且过长依赖链阻碍了下次循环迭代中的指令进入乱序发射队列，无法使能双发射，因此可通过将loop进行横向切分来降低依赖链长度。例如

```
__VEC_SCOPE__ {
    for(){
        vlds;
        op1;
        op2;
        op3;
        ...
        opn;
        vsts
    }
}

```
横向切分为：

```
__VEC_SCOPE__ {
    for(){
        vlds;
        op1;
        op2;
        op3;
        ...
        op6;
        vsts
    }
    mem_bar()
    for(){
        vlds;
        op7;
        op8;
        op9;
        ..
        opn;
        vsts
    }
}

```
通过此方法可降低每个loop中的依赖链长度，尽量使能双发射，但是缺点在于：1.可能会增大UB压力，因为会将中间结果写出到UB，所以使用此方法时需要尽量覆盖旧有UB空间。2.可能需要在loop间插入mem_bar指令保证结果的正确性，但是会增加流水线中空泡。这种横向loop切分性能收益点在于缩短了loop内的依赖链长度，使得乱序发射的有限指令窗口能够看到下次循环的无依赖关系指令，可以更好利用乱序结构使能双发射。因此其收益点与unroll展开有冲突，且循环的头尾开销，可能要插入的mem_bar往往导致该优化可能竞争不过unroll展开。但对于一些特殊情况，仍具有尝试的必要，例如算子具有明显的分段行为，前段多分支多条链，后端仅有主分支，一条链，此时对算子进行总体的unroll，性能可能不如将算子前后端分离，使能各自最优unroll。

#### loop纵向切分

loop纵向切分主要针对于循环次数进行切分，尤其对于一些循环次数为质数（例如循环13次，或43次等）或没有较小因子，unroll无法有效使能时采用。主要思想是将一个循环N次的loop拆分为循环N1次与N2次的loop(N1+N2=N)。例如：

```
__VEC_SCOPE__ {
    for(i=0; i<N; i++){
        vlds;
        op1;
        op2;
        op3;
        ...
        opn;
        vsts
    }
}

```
纵向切分为：
```
__VEC_SCOPE__ {
    #pragma unroll()
    for(i=0; i<N1; i++){
        vlds;
        op1;
        op2;
        op3;
        ...
        opn;
        vsts
    }
    #pragma unroll()
    for(i=0; i<N2; i++){
        vlds;
        op1;
        op2;
        op3;
        ...
        opn;
        vsts
    }
}

```

由于这两次循环是在处理不同地址数据，因此不需要插入mem_bar等指令。但是需要注意每个循环的起始地址，保证计算精度。该切分好处在于配合unroll，将unroll提升不明显的loop bound N拆分为两个unroll有明显提升的loop bound N1和N2。经测试，当前cce编译器支持对于任意数字的unroll，当unroll不为循环次数因子时，编译器会进行自动的loop竖向切分。例如，对于97次循环，使能`#pragma unroll(4)`编译器会自动将97次循环切分为96+1，对前96次循环进行4次unroll。但是编译器并未尝试更多可能性，例如97=16+81, 97=36+36+25等更多切分与unroll方式。为避免过大的参数搜索空间，可进行如下假设：
- 在unroll探索时，最大探索到unroll=8，
- 仅对具有较少1到8之间因子的循环次数进行竖向loop切分探索
- 切分时，subloop的循环次数之和必须为总的循环次数
- 切分后具有同样循环次数的subloop在unroll探索时应设置相同的unroll值
- subloop的数目不能超过3

### 指令重排序

指令重排序的优势在于合理安排指令顺序，尤其是长短指令顺序，尽可能降低前后指令的iniation interval时间，使能双发射，提高IPC。但需注意重排序需严格保持指令的拓扑依赖关系，否则会导致精度问题。对于如下情况往往需要进行指令重排序：
- 算子实现中，某一段具有大量的长指令(vexp，vdiv，vln，...)，规约类指令(vcmax，vcmin，vcadd，...)，以及特殊操作指令(vdup，...)等，即在仿真模型VfSim里configs/isa.json中通过SFU以及EXU0标记的复杂指令和仅能在EXU0中运算的指令。可将循环中其余位置处的短指令(vadd，vmul，vadds，...)调整到该处，降低长指令和规约类等指令密度，尽可能使能双发射，减少EXU中指令执行间隔。
- 当进行unroll展开后，unroll会将大量指令重复copy后排列，造成部分区域内长指令与规约类指令密度增大，可参考上述一点，进行指令重拍，降低长指令与规约类等特殊指令密度。

### 指令替换

规约类指令只能在EXU0中执行，并且往往需要跟随VDUP等特殊指令。过多的规约类指令会降低算子性能。因此可在算法实现层面上尽量减少规约类指令数目，例如使用(vmax vmin vadd)平替部分(vcmax vcmin vcadd)等指令。
